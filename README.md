# 1C-ULA — Universal Log Analyzer for 1C

Инструмент анализа технологического журнала платформы **1С:Предприятие**.  
Принимает файлы логов и формирует структурированный отчёт с перечнем событий, выявленных проблем и рекомендаций по их устранению.

## Возможности

- **Автоматическое определение типа лога** по имени файла и содержимому.
- **Разбор строк** двух форматов: технологический журнал (ТЖ) и старый формат `{YYYYMMDDHHmmss,...}`.
- **Классификация событий** по 8 категориям проблем (ошибки, deadlock, медленные SQL, ожидание блокировок, долгие транзакции, проблемы памяти, медленные вызовы, предупреждения).
- **Генерация рекомендаций** из встроенной базы знаний — одна рекомендация на тип проблемы.
- **Экспорт в JSON** — машиночитаемый отчёт со сводкой, списком событий, проблем и рекомендаций.
- **Экспорт в HTML** — самодостаточный отчёт с карточками сводки, таблицей событий и блоком рекомендаций.
- **Графический интерфейс** на базе tkinter — выбор файла, запуск анализа в фоновом потоке, сохранение/открытие отчётов.
- Работает **без внешних зависимостей** (только стандартная библиотека Python).
- Обрабатывает **10 000 строк менее чем за 3 секунды**.

## Поддерживаемые типы логов

| Тип       | Описание                                          |
|-----------|---------------------------------------------------|
| `rphost`  | Рабочий процесс платформы 1С                      |
| `rmngr`   | Менеджер кластера серверов 1С                     |
| `ras`     | Сервер администрирования 1С (ras)                 |
| `dbgsrv`  | Отладочный сервер платформы 1С                    |
| `1cv8`    | Клиентское приложение 1С (старый формат 1cv8.log) |

Тип определяется сначала по имени файла, затем по содержимому первых 30 строк.

## Быстрый старт

**Требования:** Python 3.8+

### CLI-режим

```bash
# Анализ лога с генерацией отчётов
python main.py rphost_24010112.log --json report.json --html report.html

# Только вывод в консоль (без сохранения отчётов)
python main.py rphost_24010112.log
```

**Пример вывода в консоль:**
```
Анализируется: rphost_24010112.log
  Тип лога:    rphost
  Событий:     1 523
  Проблем:     17
  Время:       0.142 с
  JSON-отчёт: report.json
  HTML-отчёт: report.html
```

### GUI-режим

```bash
# Запуск без аргументов открывает графический интерфейс
python main.py
```

В окне приложения:
1. Нажмите **Обзор…** и выберите файл лога (`.log`).
2. Нажмите **Анализировать** — анализ выполняется в фоновом потоке.
3. После завершения используйте **Сохранить JSON**, **Сохранить HTML** или **Открыть HTML** для просмотра результатов.

## Архитектура и модули

### Конвейер обработки

```
Файл лога
    │
    ▼
Core.Detector        ← определяет тип лога по имени / содержимому
    │
    ▼
Core.Parser          ← построчный разбор → список событий (dict)
    │
    ▼
Core.Classifier      ← присваивает категории и уровни серьёзности
    │
    ▼
Core.KnowledgeBase   ← формирует уникальные рекомендации
    │
    ├──► Report.JSONBuilder → JSON-строка
    └──► Report.HTMLBuilder → HTML-строка
```

### Описание модулей

| Модуль | Ответственность |
|--------|----------------|
| `Core.Detector` | По имени файла (regex) и содержимому (первые 30 строк) возвращает строку типа лога или `"unknown"`. |
| `Core.Parser` | Разбирает строки в два формата (ТЖ и старый). Возвращает список словарей с полями `timestamp_str`, `pid`, `event_type`, `duration_ms`, `properties`, `format`. Поддерживает потоковую обработку (`parse_log_file_iter`). |
| `Core.Classifier` | По типу события и его свойствам формирует список проблем с полями `type`, `severity`, `description`, `event`. Уровни серьёзности: `critical`, `high`, `medium`, `low`. |
| `Core.KnowledgeBase` | Возвращает по одной рекомендации (заголовок, описание, список действий) для каждого уникального типа проблемы. |
| `Report.JSONBuilder` | Формирует JSON-отчёт: `summary`, `events`, `problems`, `recommendations`. |
| `Report.HTMLBuilder` | Формирует самодостаточный HTML с CSS и JS: карточки сводки, таблица событий (первые 100 + кнопка «показать ещё»), список проблем по убыванию серьёзности, блок рекомендаций. |
| `UI.Form` | Класс `App(tk.Tk)` — главное окно. Анализ запускается в отдельном `threading.Thread`, чтобы не блокировать интерфейс. |

## Форматы входных данных

### Технологический журнал (ТЖ)

```
MM:SS.microseconds-PID,EVENT,duration_ms[,key=value,...]
```

Пример:
```
12:34.567890-1234,SDBL,1200,Sql="SELECT * FROM ...",Rows=500
```

### Старый формат (`1cv8.log`)

```
{YYYYMMDDHHmmss,LEVEL,AppName,Process,Description}
```

Пример:
```
{20240101123456,ERR,1cv8,rphost,Ошибка выполнения запроса}
```

## Классификация событий

| Тип проблемы | Серьёзность | Условие |
|---|---|---|
| `error` | critical | Событие `EXCP` или `ERR` |
| `deadlock` | critical | Событие `TDEADLOCK` |
| `lock_wait` | high | Событие `TLOCK` |
| `long_transaction` | high | Событие СУБД (`DBPOSTGRS`, `DBORACLE`, `DBMSSQL`, `DBMSSQLCONN`) и длительность > 2 000 мс |
| `memory_issue` | high | Событие `MEM` и Memory > 1 ГБ |
| `slow_sql` | medium | Событие `SDBL` и длительность > 500 мс |
| `slow_call` | medium | Событие `CALL` и длительность > 10 000 мс |
| `warning` | low | Событие `WARNING` |

## Формат отчётов

### JSON

```json
{
  "summary": {
    "log_type": "rphost",
    "total_events": 1523,
    "total_problems": 17,
    "counts_by_event_type": { "SDBL": 300, "CALL": 50, ... },
    "counts_by_problem_type": { "slow_sql": 12, "error": 5, ... },
    "analysis_time_sec": 0.142
  },
  "events": [ { "timestamp_str": "...", "event_type": "...", ... } ],
  "problems": [ { "type": "slow_sql", "severity": "medium", "description": "...", ... } ],
  "recommendations": [ { "problem_type": "slow_sql", "title": "...", "actions": [...] } ]
}
```

### HTML

Самодостаточный файл без внешних зависимостей:
- **Карточки сводки** — тип лога, количество событий, разбивка по типам проблем, время анализа.
- **Таблица событий** — время, тип, длительность, PID, описание. Первые 100 строк видны сразу, остальные раскрываются кнопкой.
- **Список проблем** — отсортирован по убыванию серьёзности, с цветными бейджами.
- **Блок рекомендаций** — конкретные шаги по устранению каждого типа проблемы.

## Зависимости

Проект использует **только стандартную библиотеку Python**. Внешняя зависимость одна — `pytest` для запуска тестов.

| Пакет | Назначение |
|-------|-----------|
| `pytest` | Запуск модульных тестов |

GUI требует наличия модуля `tkinter` (входит в стандартную поставку CPython; на некоторых дистрибутивах Linux устанавливается отдельно: `sudo apt install python3-tk`).
